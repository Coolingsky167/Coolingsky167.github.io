简答题

异常处理

  • 关于 Web service 的 Java 的异常应该被转化成 WSDL 的 fault（wsdl:fault）
  • 关于运行时的动态异常则不应该被转化成 fault
  • 使用 @WebFault 注解
  • 如果想自定义错误处理流程，可以使用 JAX-WS 的 handlers

WSDL Message Dispatch

  • 如果端点地址只和wsdl中的一个service关联，那么指定唯一的Qname作为输入就足够了

  • 接收到消息后，消息应该被发送到正确的服务。使用xsd作为type system的时候，一些“#any”,”#none”的token会导致歧义，可以通过将wsdl:required设置为true

  • WS寻址提供了一种消除歧义的机制：提供了默认的Action，可以唯一标识每种消息类型，也可以定义额外的元素。

  • 当使用HTTP绑定或使用带有SOAP Response MEP的SOAP绑定，因为request消息中没有SOAP envelope，所以上述不可行，不过也可以构造{address}和{http location}来提供可以与操作唯一关联的位置

分发处理的步骤？
  1. 识别soap消息的目标soap:address
  2. 从soap:address得到wsdl:port和wsdl:portType
  3. 识别它的目标wsdl:operation
  4. 查找相关联的java类和方法

问答题

1.根据Java Web Service的调用模型，结合下图，举例阐述在server端和Client端的调用过程

服务端调用

  1. 接受来自网络传输层(例如，HTTP 或JMS端点)的SOAP消息。
  2. 调用处理器(handler)来对消息进行预处理(例如，为可靠性的目的而对消息进行持久化处理，或者处理SOAP消息的标头)。
  3. 决定消息的目标服务，换句话说，消息应该调用哪个WSDL操作
  4. 在确定了目标WSDL操作后，决定调用哪个Java类方法。将确定的Java类/方法称为Java目标。决定Java目标的过程称为分发(dispatching)
  5. 把SOAP消息传递给序列化(Srializatio)子系统，将它反序列化为Java对象，这些Java对象将作为参数传递给Java目标。
  6. 使用序列化子系统生成的参数来调用Java 目标，而后得到目标方法返回的Java对象。
  7. 把返回的Java对象再传递给序列化子系统，将它序列化为与目标WSDL操作指定的返回消息一致的 XML元素。
  8. 把返回的XML元素包装成与目标WSDL操作相一致的SOAP响应消息。
  9. 把SOAP响应消息传回给网络传输层以发送消息。

客户端调用

  1. 创建一个Web服务端点的实例，该服务端点实现服务端点接口(Service Endpoint Interface, SEI)。调用子系统有一个或多个工厂方法可以用于创建SEI实例。我们可以方便地创建这些实例，也可以用JNDI来访问它们。通常情况下，SEI 实例是用Java代理和调用处理器(Invocation Handler)来实现的。

  2. 处理 SEI实例的调用。

  3. 获得传递给 SEI的参数，把它们传递给序列化子系统，然后序列化为与目标服务的WSDL指定的XML模式一致的 XML元素。

  4. 基于目标服务的WSDL,把参数元素封装到SOAP消息中。

  5. 调用处理器， 它基于服务质量(Quality of Service, QoS)和其他需求对消息进行后续处理(例如，为可靠性的目的而对消息进行持久化处理，或者设置SOAP标头)

  6. 把消息传递给网络传输， 向目标Web服务交付消息。

  7. 接收来自网络传输的SOAP消息响应。

  8. 把SOAP响应消息传递给序列化子系统，把它反序列化为Java对象，该Java对象是SEI的返回类型指定类的实例。

  9. 返回反序列化的 S0AP响应消息，完成SEI的调用。

2.两个图分别显示了Java Web Service在server和client端的调用模型，试结合两图，举例阐述在Server端和Client端的调用过程

  1. JWS 中，Port 组件(component, 有时也称为Port)是一个Web服务的服务器视图。可以把一个Port 组件打包为一个WAR (用于部署为一个servlet 端点)或一个EJB JAR (用于部署为一个EJB端点)。WSEE]是用于定义部署过程和打包结构的主要标准。WS-Metadata描述了如何通过标注(annotation) 来部署需要打包的类(例如，可以用WS-Metadata标注来定义代表一个Port组件的WSDL接口)。部署一个Java EE 5 Port组件的功能相当于在WSPA参考系统中部署一个子系统的功能。
  2. 转移到图2-2 的顶部，第1步描述了通过Port组件可以部署Web服务，现在沿着对部署好的Web服务的调用而展开描述。服务端点(endpoint)支持通过HTTP GET来请求描述Web服务的WSDL文档。这一WSDL的结构是由JAX-WS WSDL/Java映射决定的。可以使用由JAX-WS、JAXB,以及WS-Metadata定义的标注来定制这些映射。
  3. 在服务端点接收到一个通过HTTP POST发送来的SOAP请求时，就会引发部署在这个端点上的Web服务的调用。
  4. Web服务的端点通常是通过servlet来实现的，这个servlet可以监听部署服务(参见步骤1)时指定的端点URL。这个servlet不是由Port组件的开发人员指定的，而是Java EE 5内部的JWS实现的一部分。在部署Web服务时，WSEE运行时引擎会读取标注和(或)部署描述符中的配置信息，以决定服务端点的URL,并在容器内部进行配置，以便为此端点URL部署相应的监听器(listener) 。建立服务端点是WSPA中部署子系统的功能。
  5. 第5步显示SOAP请求已经被转换为一组XML参数，在 WSDL中指定为参数的XML Schema组件的实例。JAX-WS运行时引擎负责从SOAP消息中提取这些参数。在WSPA的调用子系统中，这一XML参数提取过程被定义为分发处理(dispatching process)的一部分。

  1. 接下来，XML参数被反序列化(eseraire)) 为Java参数。这一反序列化过程由JAXB运行时引擎进行处理，可以通过目标Java 类中的标注对反序列化加以控制(也就是定义类型映射)。JAXB运行时引擎使用内省(introspectio) 机制来检查目标类中的标注，再加上由JAXB定义的标准的Java/XML绑定信息，就可以生成目标参数的实例了。JAXB序列化和相关的标注将在第5章加以介绍。这一步相当于WSPA中的序列化子系统。
  2. 在创建好了参数以后，就可以调用目标Java类/方法。在WSPA中，这一功能由调用子系统负责处理。在JavaEE5中，由JAX-WS运行时引擎来处理调用管理。提供调用方法的类可以是EJB,也可是POJO。如图2-2所示，Web服务应用程序是由经过标注的Java类组成的，JAX-WS、JAXB,以及WS-Metadata在这些类的定义中均扮演了一定角色，这些类都使用了相关JSR 中的标注。必须使用WS-Metadata标注(如@webService )来标明被调用的类(被称为服务实现bean)，
    以表明它实现了一个Web服务，再用JAXB标注来标明这些类方法的参数和返回值。此外，对于服务实现bean, 还可以用一 些JAX-WS标注来控制WSDL的样式
  3. 在调用之后， 处理过程将按与以上步骤相反的方向进行处理。Java方法返回的数据类型由JAXB进行序列化处理，以转换成XML返回值的类型的实例。这种序列化(与第6步中描述的反序列化类似)是通过对Java返回类上添加的标注来加以控制的。
  4. 接下来， JAX-WS运行时引擎将收到的XML返回值的类型的实例封装到SOAP响应中。
  5. 最后，SOAP响应作为HTTP响应的有效负载而被发送回请求者。

  1. 在客户端调用Web服务的第一步， 通常是使用某种WSDL到Java的映射工具来生成一个服务端点接口(Service Endpoint Interface, SEI。SEI 提供了对需要调用的Web服务的Java表示。在编译客户端应用程序之前，就应该先生成SEI,因为在客户端应用程序中要使用它的服务接口定义。生成SEI要使用到JAXB、JAX-WS、WS-Metadata
  2. 在运行时，使用JAX-WS的javax. xml .ws . Service类的一个实例来提供被调用的Web服务的客户端视图，用一个Service .getPort方法来获取上一步中生成的SEI的运行时实例。这个运行时实例是使用Java代理(proxy) 技术而实现的，在图2-3中表示为个Proxy 实例(instance) 。如图2-3 所示，这一个Proxy实例实现了服务端点接口(SEI)。
  3. 接下来，通过调用Proxy 实例上的一个SEI方法，就可以调用Web服务。传递给SEI方法的参数是在创建SEI (第1步)时生成的类的实例。这些类是JAXB根据WSDL中包含的模式而映射定义的。如图2-3所示，由Proxy实例实现的SEI也使用了WS-Metadata和JAX-WS。在SEI中会使用一些由这些 JSR定义的标注，不过，这些标注不会影响用户对SEI方法的调用。JAX-WS运行时引擎会用这些标注把JAXB参数和方法调用转换成要向Web服务发送的SOAP请求。JAX.WS运行时引擎生成好SOAP请求以后，在将它发送到Web服务端点之前，会先调用处理器(如果存在)。处理器(handler)是由用户定义的，可以对SOAP请求/响应的消息进行处理。WSEE规范定义了封装客户端处理器的过程，JAX-WS运行时引擎负责对处理器进行调用。WS-Metadata 规范也定义了用于将处理器附加到端点实现的标注。
  4. 在处理 器完成处理之后，SOAP请求消息就会被发送到Web服务，Web服务再返回一个 SOAP响应。SOAP响应的处理过程顺序与前面的顺序相反。首先应用处理器，接着将SOAP响应参数进行反序列化(deserialized) (通过 JAXB运行时引擎)。
  5. 最后， Proxy 实例返回一个JAXB实例作为调用的返回值。

3.结合JAX-WS，JAXB，WSEE，WS-Metada等JSR，讨论如何在Java Web Service Platform中使用 start from WSDL , start from Java , start from WSDL and Java 三种开发方式并且对比这三种开发方式

  1. JWS旨在帮助Java开发者更容易地开发Web服务。因此，它偏重于 “从Java开始 “的开发模式。

    1. “从 Java 开始 “假设从开发一个 Java 类开始创建 Web 服务。
      在大多数情况下，就是用 @WebService 来注释该类，以便将其部署为 Web 服务。JAX-WS 和 JAXB 运行时引擎将使用标准的 WSDL/Java 和 XML/Java 映射法将该类映射为 WSDL 文档。

    2. 在 “从Java开始 “的开发模式下，如果需要调整你的 WSDL 或 XML 类型，可以通过向类添加注解来完成。”从Java 开始”是WS-Metadata实现所要求的唯一开发模式。

      另外，JWS也定义了 “从WSDL开始 “的开发模式。

    3. 使用这种方法，你需要一个预先存在的 WSDL，并使用 WSDL 编译器（作为 JAX-WS 实现的一部分提供）来生成一组实现 WSDL 的 Java 程序元素。

    4. 使用这种方法，你可以得到一组与 WSDL 对应的 SEI。你的工作是用适当的业务逻辑实现 SEI，以实现 WSDL 定义的 Web 服务。

  2. 在 “从 WSDL 和 Java 开始 “的开发模式下，你有预先存在的 XML schema，必须与预先存在的 Java 对象（参数和返回类型）相对应。这些如果没有一个高度可定制的序列化子系统，这些问题就无法得到优雅的解决。

  3. 因此，几乎所有试图简化序列化问题的 Web 服务平台（例如，通过定义标准的序列化上下文，如 JAX-RPC 1.1）最终都会引入对包装代码的需求，以解决 “从 WSDL 和 Java 开始 “的问题。

    1. 开发者使用 JAX-WS WSDL 编译器，按照从 WSDL 开始的开发模式，从现有的 WSDL 中生成一个 SEI。然后，这个 SEI 通过包装器被映射到现有的类中。

  4. 另一方面，有一种情况是，从 WSDL 和 Java 开始的开发模式实际上是最容易处理的，因为当 WSDL 和 Java 都被定义时，开发者不需要做任何设计工作。也就是说，开发者不需要设计一个灵活而有用的WSDL或一个好的Java API。

从设计者的角度看这可能是真的，但从开发者的角度看，”从WSDL和Java开始是使用JWS工具最难解决的问题。JWS 可以从你的 Java 中生成有效的 WSDL，它可以从你的 XML Schema 中编译 Java 类，但是当涉及到将现有的 Java 映射到现有的 XML Schema 和 WSDL 时，它不能帮你太多。
